# PRODUCTION OPERATIONS MANUAL - Mana Events Marketplace

## 1. ARCHITECTURE OVERVIEW
Mana Events Marketplace is built on a Modular Monolith architecture using Next.js, Prisma, and Inngest.

- **Frontend**: Next.js App Router (React 19)
- **Backend**: Next.js API Routes (Edge & Node.js runtimes)
- **Database**: PostgreSQL (Prisma ORM)
- **Caching/Queue**: Redis (Upstash)
- **Background Jobs**: Inngest
- **Search**: Meilisearch
- **Real-time**: Socket.IO

## 2. MONITORING & OBSERVABILITY

### Health Checks
- **Endpoint**: `/api/health`
- **Monitored Services**: Database, Redis, Meilisearch, Socket.IO, Inngest, Cloudinary, Resend, Twilio.
- **System Metrics**: CPU Load, Memory Usage, Uptime, Response Time.

### Logging (Winston)
Structured logging is implemented via `src/lib/logger.ts`.
Every log includes:
- `requestId`, `correlationId`, `userId`, `vendorId`, `bookingId`, `environment`.

### Metrics (OpenTelemetry)
Metrics are collected via OpenTelemetry and can be exported to any OTLP-compatible backend (Honeycomb, Prometheus, etc.).
Key Metrics:
- `api_latency`: API response times.
- `errors_total`: Error rates by service and code.
- `bookings_total`: Throughput of bookings.
- `payments_total`: Payment success/failure rates.
- `active_connections`: Live Socket.IO connections.

## 3. ERROR TRACKING
- **Sentry**: Integrated for unhandled exceptions and performance profiling (Optional via `SENTRY_DSN`).
- **Alert Engine**: Automated alerts for critical failures (Database down, Slow APIs).

## 4. SECURITY OPERATIONS
- **Rate Limiting**: IP-based and User-based rate limiting on all sensitive endpoints.
- **Account Lockout**: Users are locked out after 5 failed login attempts for 15 minutes.
- **Audit Trails**: Critical actions (Logins, Payments, Refunds) are recorded in the `AuditLog` table.
- **Environment Validation**: `validateEnv()` ensures all required secrets are present at startup.

## 5. INCIDENT RESPONSE
1. **Detection**: Monitoring alerts (Slack/Email) or Health Check failure.
2. **Analysis**: Check Logs in Winston/Sentry.
3. **Containment**: Scale services, rotate keys, or enable maintenance mode if necessary.
4. **Recovery**: Follow the [Restore Procedure](#7-recovery-procedures).

## 6. BACKUP & RETENTION
- **Database**: Daily automated backups with 30-day retention.
- **Verify Integrity**: Monthly restore tests are performed by the DevOps team.

## 7. RECOVERY PROCEDURES

### Database Restore
1. Identify the latest healthy snapshot.
2. Restore to a staging environment first to verify data.
3. Update `DATABASE_URL` to point to the restored instance.

### Rollback Plan
- **Vercel**: One-click rollback to the previous successful deployment.
- **Database**: If migrations failed, run `prisma migrate resolve --rolled-back <migration_name>`.

## 8. SCALING GUIDE
- **Horizontal Scaling**: The Next.js app is stateless and can be scaled across multiple regions.
- **Database Scaling**: Use read-replicas for heavy read traffic.
- **Redis**: Scale Upstash Redis tier based on command volume and memory usage.
